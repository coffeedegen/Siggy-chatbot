SECTION 4: EVM++ — ENHANCED ETHEREUM VIRTUAL MACHINE
=============================================================

EVM++ is a backward-compatible extension of the Ethereum Virtual Machine. Developers can use the same tooling they already know while accessing Ritual's advanced features.

Account Abstraction via EIP-7702:
Ritual is one of the earliest chains to support EIP-7702, proposed by Vitalik Buterin. It introduces a new transaction type called SetCodeTx, which allows EOA accounts to delegate a smart contract as their implementation.

This enables:
- Batch calls: Send multiple transactions at once.
- Gas fee sponsorship: Frictionless zero-fee user transactions.
- Arbitrary signing keys: Support for P256, BLS, and other signature schemes.
- Advanced access control lists: Granular usage permissions and expiring keys.

Supported EIP Extensions:
- EIP-7702: Set EOA account code. Enables native account abstraction without wallet migration.
- EIP-7212 / RIP-7212: secp256r1 (P-256) signature verification. Enables Passkeys, Apple Secure Enclave, and hardware signing keys.
- EIP-5920 (PAY): New opcode to send ETH without triggering callee code. Gas-efficient ETH transfers without side effects.

SECTION 5: EXECUTION SIDECARS
=============================================================

What are Sidecars?
Unlike Ethereum, which implements precompiles directly in execution client code, Ritual implements its compute precompiles as modular processes (sidecars) that run parallel to the execution client. Any developer who can build a containerized sidecar can have their computation enshrined on-chain.

Supported Sidecar Types:

Classical ML Inference (ONNX)
Description: Canonical tree-based and regression-based models via ONNX format. Sourced from HuggingFace or Arweave.

LLM Inference
Description: Large language models of any size and architecture. Supports TGI, llama.cpp, and ONNX engines across CPU, GPU, NPU, and Apple Silicon.

ZK Proving and Verification
Description: Generate ZK proofs for AI inference. Proof-system agnostic — supports RISC Zero and others. Abstracts circuit complexity.

TEE Execution
Description: Secure computation inside Trusted Execution Environments for privacy-preserving inference, autonomous agents, and external API calls.

Chain Abstraction
Description: Natively read and write state to and from other blockchain networks via Wormhole queries.

Network Calls
Description: HTTP requests executed inside TEE enclaves with cryptographic attestation. Connects smart contracts to any web API or service.

TEE vs ZK Proofs Comparison:
- TEE Performance: High, suitable for real-time applications.
- ZK Proof Performance: High overhead, can take minutes or hours.
- TEE Security Basis: Hardware-based (Intel SGX, ARM TrustZone).
- ZK Proof Security Basis: Pure cryptographic guarantees.
- TEE Trust Assumptions: Requires trust in chip manufacturer.
- ZK Proof Trust Assumptions: No hardware trust required.
- TEE Best Use: AI inference, autonomous agents, external API calls.
- ZK Proof Best Use: Proof of model authenticity, verifiable inference.

SECTION 6: RESONANCE — FEE MECHANISM
=============================================================

What is Resonance?
Resonance is a general two-sided fee market that matches users (transactions) to nodes. It is surplus-maximizing, meaning the winning allocation is always the most economically efficient.

How it works for Users:
- Specify a private valuation (maximum willingness to pay).
- Can prioritize speed or cost.
- Can execute any compute type.
- Private execution via multi-party assignment.

How it works for Nodes:
- Specify a private cost function per transaction type.
- Can specialize in specific compute such as AI inference.
- Execute non-conflicting transactions in parallel.

How it works for the Network:
- Transactions efficiently matched to optimize cost and speed.
- Node capacity optimally allocated.
- No protocol-level compute constraints.

Key Actors in Resonance:

Brokers:
Brokers are sophisticated profit-seeking agents that compute efficient transaction-to-node matchings, pocketing the spread between user valuation and node execution cost. They are analogous to MEV searchers or block builders in Ethereum.

Auctioneer:
The Auctioneer is the central orchestrator with two components:
1. An on-chain smart contract (Auctioneer.sol) managing node registration and payment settlement.
2. An off-chain service that runs Resonance batches, decrypts valuations, and selects the surplus-maximizing routing.

Resonance is fully opt-in alongside the standard EIP-1559 fee model. Users upgrade their wallet via SetCodeTx to AuctioneerWallet.sol.

SECTION 7: SYMPHONY — CONSENSUS PROTOCOL
=============================================================

What is Symphony?
Symphony is Ritual's new consensus protocol implementing Execute-Once-Verify-Many-Times (EOVMT). Instead of requiring every node to re-execute every transaction, selected compute nodes act as sole executors and generate succinct sub-proofs that all other validators verify efficiently.

Key Properties:
- Inference SPC opcodes execute asynchronously off-chain by designated nodes.
- Proof generation occurs off-chain, not on the critical path for block production.
- Ritual prioritizes prover-friendly systems with linear or near-linear proof generation time.
- Parallel proof generation via multi-core or multi-machine setups is supported.

Dual Proof Sharding:
Symphony partitions proof verification across node subsets (sharding), enabling the network to verify large computational outputs without requiring every node to check every proof. Attested committees validate sub-proofs, and distributed verification ensures security while dramatically improving scalability.

SECTION 8: SCHEDULED TRANSACTIONS AND ORACLES
=============================================================

What are Scheduled Transactions?
Scheduled transactions enable recurring, conditional invocation of smart contract functions at the top of a block, without external keepers.

Key advantages:
- Priority execution at top-of-block — critical for oracle updates before user deposits and withdrawals.
- Predictable update intervals for automated recurring operations.
- Guaranteed execution with separately priced inclusion fee markets.

Verifiable Oracle Types:

Model Execution Oracles:
Execute classical ML inference at fixed frequencies using on-chain and off-chain data to update DeFi protocol parameters such as liquidation thresholds.

TEE-Based Data Oracles:
Collect off-chain data at fixed intervals via scheduled transactions. Data is validated by TEE attestations and enshrined on-chain at top-of-block.

Infernet Integration:
Infernet is a decentralized oracle network purpose-built for AI and heterogeneous compute with 8,000+ independent nodes. With native integration into Ritual Chain, Infernet nodes act as compute coprocessors for long-running tasks and async execution.

SECTION 11: VERIFICATION AND PRIVACY
=============================================================

Ritual is proof-system agnostic, allowing developers to choose the right verification strategy for their application.

Verification Methods:

ZKML (Zero-Knowledge ML):
Mechanism: Zero-knowledge proofs for ML.
Best For: Maximum security and privacy. Verifiable inference without revealing inputs or weights.

OPML (Optimistic ML):
Mechanism: Assume correct, then challenge period.
Best For: Efficient verification of large models where interaction is acceptable.

PPML (Probabilistic Proof ML):
Mechanism: Backdoor-based statistical verification.
Best For: Cheap, non-interactive verification of model provenance.

TEE (Trusted Execution Environment):
Mechanism: Hardware-based secure enclaves.
Best For: Real-time applications, privacy-preserving inference, GPU workload support via CPU-TEE bridging.

Verifiable Provenance:
Ritual maintains immutable records of model origin, updates, and transformations. This includes verifiable model training, protection against model forgery, and preservation of inference quality — all backed by on-chain cryptographic proofs.

Modular Storage:
- HuggingFace Hub (Web2): Pre-trained ONNX models and large model files.
- Arweave (Web3): Permanent, decentralized storage for models and provenance records.

SECTION 16: GLOSSARY
=============================================================

Blockchain: A decentralized, append-only database maintaining a cryptographically secured chain of blocks. Each block is linked to its predecessor through cryptographic hashing.

Smart Contract: Self-executing programs on a blockchain that automatically enforce and execute agreements without intermediaries.

L1 (Layer 1): A base blockchain network like Ethereum or Ritual Chain that handles its own transaction validation and security.

EOA: Externally Owned Account. A regular blockchain account controlled by a private key such as a personal wallet.

Staking: Locking up tokens as collateral to participate in network consensus or validation. Staked tokens can be slashed for malicious behavior.

ZK Proof: Zero-Knowledge Proof. A method to prove something is true without revealing the underlying information.

EVM: Ethereum Virtual Machine. A Turing-complete state machine executing smart contract bytecode deterministically across all network nodes.

EVM++: Ritual's enhanced EVM adding native AI operations, scheduled transactions, and advanced features while maintaining backward compatibility.

Symphony Protocol: Consensus protocol implementing dual proof sharding, distributed verification, and the execute-once-verify-many-times paradigm.

Resonance Mechanism: Dynamic fee mechanism that matches compute supply with demand through a surplus-maximizing market-driven approach.

EOVMT: Execute-Once-Verify-Many-Times. An optimization where one node executes complex computations and others verify via cryptographic proofs.

Execution Sidecar: Modular component handling specialized computation tasks asynchronously from the main chain, enabling complex operations with EVM compatibility.

Enshrined Model: An AI model natively represented on-chain as a first-class citizen, enabling direct smart contract integration and verifiable execution.

Node Specialization: Architecture allowing nodes to optimize for specific computation types such as AI, ZK, or TEE while still participating in network consensus.

Proof Sharding: Symphony technique distributing proof verification across different node subsets for improved scalability and efficiency.

Infernet: Decentralized oracle network with 8,000+ nodes, purpose-built for bringing verified AI computation on-chain across any EVM-compatible blockchain.

Ritual Guardians: Specialized nodes monitoring network health and verifying complex computations. Includes a firewall system for selective execution participation.

Ritual Portals: Cross-chain bridges on GMP protocols enabling secure access to Ritual's compute from any blockchain.
