Executive Summary

Ritual is a sovereign Layer 1 blockchain purpose-built for heterogeneous
compute — with a primary focus on bringing AI natively on-chain. Unlike
existing blockchains that optimize for transaction throughput, Ritual
reimagines what on-chain computation can mean: enabling AI inference,
zero-knowledge proofs, trusted execution environments (TEEs), and
scheduled transactions as first-class primitives. Its core thesis is
that enriching what users can do on-chain today is the only path to
attracting the users of tomorrow.

Ritual's architecture is powered by three novel innovations: EVM++ (a
backward-compatible extension of the Ethereum Virtual Machine),
Resonance (a surplus-maximizing fee mechanism for heterogeneous
compute), and Symphony (a new consensus protocol enabling
execute-once-verify-many-times semantics). Together, these make Ritual
the schelling point for web3 developers who need expressive computation
beyond simple token transfers.

1. What Is Ritual?

1.1 Core Mission

Ritual was born from a single focus: enrich what users can do on-chain
today to attract the users of tomorrow. The project began at the
intersection of Crypto and Artificial Intelligence, making smart
contracts actually smart by allowing them to natively tap into on-chain
AI, backed by the same trustless properties of modern blockchains.

Ritual is not limited to Crypto x AI. By optimizing for any type of
heterogeneous compute, Ritual hyperscales expressive computation
on-chain. It is the best venue for developers building with AI, ZK
proofs, or TEEs — and for those experimenting at the edges of unknown
paradigms.

4. EVM++

4.1 Overview

EVM++ is a backward-compatible extension of the Ethereum Virtual Machine
with expressive compute precompiles, native scheduling, built-in account
abstraction, and support for in-demand EIPs. Developers can
progressively adopt Ritual Chain functionality using the same tooling
they already know.

4.2 Account Abstraction (EIP-7702)

Ritual is one of the earliest chains to support EIP-7702 (Set EOA
Account Code), proposed by Vitalik Buterin. Traditional EOA accounts are
limited to sending and receiving transactions signed by a private key.
EIP-7702 introduces a new transaction type, SetCodeTx, which allows EOA
accounts to delegate a smart contract as their implementation.

This enables smart account capabilities without a full wallet migration,
including:

-   Batch calls — send multiple transactions at once

-   Gas fee sponsorship — frictionless zero-fee user transactions

-   Arbitrary signing keys — support for P256, BLS, and other signature
    schemes

-   Advanced access control lists — granular usage permissions and
    expiring keys

5. Execution Sidecars

5.1 What Are Sidecars?

Unlike Ethereum, which implements precompiles directly in execution
client code, Ritual implements its compute precompiles as disjoint
sidecars — modular processes that run and scale parallel to the
execution client. Any developer who can architect a containerized
sidecar can have their computation enshrined on-chain.

Sidecars are currently stateless, but will soon be extended to stateful
operation, enabling asynchronous execution for long-running compute
precompiles.

5.3 AI Inference — LLM Details

The LLM landscape has shifted toward open-source dominance. Key
developments informing Ritual's approach:

-   Open-source models now match or exceed closed models in specialized
    benchmarks (e.g., DeepSeek-R1 vs. OpenAI o1 on math reasoning)

-   Small Language Models (3B parameters) achieve production-grade
    performance, enabling edge deployment

-   Multiple inference patterns are now viable: TGI for high-throughput,
    llama.cpp for edge, ONNX for standardized inference

Ritual's innovation: a Universal Inference Layer that abstracts hardware
complexity, supports any model architecture and inference engine, and
delivers verifiable execution via Symphony's dual proof sharding.

5.5 Network Call Sidecar

The Network Call Sidecar allows smart contracts to initiate arbitrary
HTTP requests as part of on-chain execution — a paradigm shift from
traditional oracle models. Requests are:

-   Routed through a precompile to the Network Call Sidecar

-   Executed within a TEE (Trusted Execution Environment)

-   Cryptographically attested and returned to the smart contract

-   Verifiable end-to-end, including encrypted credential handling

Use cases enabled: dynamic DeFi with long-tail data sources, on-chain
agents using CEX data, cross-platform Web2 API integrations, and fully
autonomous agentic workflows.

6. Resonance — Fee Mechanism

6.1 Problem: Existing Fee Mechanisms Are Insufficient

As blockchain transactions grow more heterogeneous (Bitcoin arithmetic →
Ethereum smart contracts → blobs → parallel execution → AI), existing
fee mechanisms like multi-dimensional EIP-1559 break down:

-   Poor welfare guarantees as the number of resource dimensions
    increases

-   Requires hard-coding resource types into the protocol (only
    changeable via hard forks)

-   Information-theoretic challenges in setting correct base fees across
    many dimensions

6.2 Resonance Solution

Resonance is a general two-sided fee market matching users
(transactions) to nodes. It is surplus-maximizing — the winning
allocation is always the most economically efficient. Key design
choices:

  -----------------------------------------------------------------------
  Stakeholder     Resonance Design
  --------------- -------------------------------------------------------
  Users           Specify a private valuation (max willingness to pay);
                  can prioritize speed or cost; can execute any compute
                  type; private execution via multi-party assignment

  Nodes           Specify a private cost function per transaction type;
                  can specialize in specific compute (e.g., AI
                  inference); execute non-conflicting transactions in
                  parallel

  Network         Transactions efficiently matched to optimize cost and
                  speed; node capacity optimally allocated; no
                  protocol-level compute constraints
  -----------------------------------------------------------------------

6.3 Key Actors

Brokers

Brokers are sophisticated profit-seeking agents that compute efficient
transaction-to-node matchings, pocketing the spread between user
valuation and node execution cost. Analogous to MEV searchers or block
builders in Ethereum. Resonance provides strong guarantees on broker
behavior at pure Nash equilibrium.

Auctioneer

The Auctioneer is the central orchestrator with two components: (1) an
on-chain smart contract (Auctioneer.sol) managing node registration and
payment settlement, and (2) an off-chain service that runs Resonance
batches, decrypts valuations, and selects the surplus-maximizing
routing. Currently one batch per block; streaming implementations are in
development.

6.4 Transaction Lifecycle with Resonance

Resonance is fully opt-in alongside standard EIP-1559. Users upgrade
their wallet via SetCodeTx to AuctioneerWallet.sol. The AuctioneerWallet
reverts until a match is assigned, then executes and pays the matched
node. Integration with Viem, ethers.js, and Web3.js is handled
automatically.

7. Symphony — Consensus Protocol

7.1 Problem: Replicated Execution Doesn't Scale

Traditional blockchains require every node to re-execute every
transaction. For AI inference, ZK proving, or TEE execution — which can
take minutes or hours — replicating this across all nodes is infeasible.

7.2 Symphony's EOVMT Model

Symphony implements Execute-Once-Verify-Many-Times (EOVMT). Selected
compute nodes act as sole executors of workloads, generating succinct
sub-proofs. All other validators verify these proofs efficiently,
without re-executing. Key properties:

-   Inference SPC opcodes execute asynchronously off-chain by designated
    nodes

-   Proof generation occurs off-chain, not on the critical path for
    block production

-   Ritual prioritizes prover-friendly systems with linear/near-linear
    proof generation time

-   Parallel proof generation via multi-core or multi-machine setups is
    supported

7.3 Dual Proof Sharding & Distributed Verification

Symphony partitions proof verification across node subsets (sharding),
enabling the network to verify large computational outputs without
requiring every node to check every proof. Attested committees validate
sub-proofs, and distributed verification ensures security while
dramatically improving scalability.

8. Scheduled Transactions & Enshrined Oracles

8.1 Scheduled Transactions

Scheduled transactions enable recurring, conditional invocation of smart
contract functions at the top of a block, without external keepers. Key
advantages:

-   Priority execution at top-of-block — critical for oracle updates
    before user deposits/withdrawals

-   Predictable update intervals for automated recurring operations

-   Guaranteed execution with separately priced inclusion fee markets

8.2 Verifiable Oracle Types

  -----------------------------------------------------------------------
  Oracle Type         Description
  ------------------- ---------------------------------------------------
  Model Execution     Execute classical ML inference at fixed frequencies
  Oracles             using on-chain and off-chain data to update DeFi
                      protocol parameters (e.g., liquidation thresholds)

  TEE-Based Data      Collect off-chain data at fixed intervals via
  Oracles             scheduled transactions; data validated by TEE
                      attestations and enshrined on-chain top-of-block

  Traditional Oracle  Requires external keepers, trusted validator
  (Comparison)        quorums, and lacks native verification — all
                      eliminated by Ritual's approach
  -----------------------------------------------------------------------

8.3 Infernet Integration

Infernet is a decentralized oracle network purpose-built for AI and
heterogeneous compute with 8,000+ independent nodes. With native
integration into Ritual Chain, Infernet nodes act as compute
coprocessors for long-running tasks and async execution. This enables
cross-chain job execution, delegated compute, and additional compute
backbone capacity.

-   Arweave (Web3) — permanent, decentralized storage for models and

12. Cross-Chain Capabilities

12.1 Ritual Portals

Ritual Portals are cross-chain bridges built on GMP (General Message
Passing) protocols that enable secure access to Ritual's specialized
compute from any blockchain. Portals handle message verification, state
synchronization, and result delivery.

12.2 Chain Abstraction Sidecar

The Chain Abstraction sidecar enables smart contracts on Ritual to
natively read (and soon write) state on other blockchain networks —
including MegaETH, Movement, and all major EVM chains. This allows
cross-chain yield aggregation, multi-chain agent operations, and
seamless interoperability.

13. Guardians

Guardians are specialized nodes that monitor network health and verify
complex computations. They form a security layer complementing
traditional validators, particularly for heterogeneous workloads.
Guardians include a firewall system allowing nodes to granularly opt-in
to specific types of execution while still participating in consensus.

15. Glossary

Blockchain Fundamentals

  -----------------------------------------------------------------------
  Term            Definition
  --------------- -------------------------------------------------------
  Blockchain      A decentralized, append-only database maintaining a
                  cryptographically secured chain of blocks. Each block
                  is linked to its predecessor through cryptographic
                  hashing.

  Smart Contract  Self-executing programs on a blockchain that
                  automatically enforce and execute agreements without
                  intermediaries.

  L1 (Layer 1)    A base blockchain network like Ethereum or Ritual Chain
                  that handles its own transaction validation and
                  security.

  EOA             Externally Owned Account — a regular blockchain account
                  controlled by a private key (e.g., a personal wallet).

  Staking         Locking up tokens as collateral to participate in
                  network consensus or validation. Staked tokens can be
                  slashed for malicious behavior.

  ZK Proof        Zero-Knowledge Proof — a method to prove something is
                  true without revealing the underlying information.
  -----------------------------------------------------------------------

Key Takeaways

-   Ritual is the first blockchain purpose-built for heterogeneous
    compute, making AI inference, ZK proofs, and TEE execution native
    first-class operations rather than afterthoughts.

-   Three novel innovations define Ritual's architecture: EVM++
    (expressive compute precompiles + account abstraction + scheduling),
    Resonance (surplus-maximizing fee market for heterogeneous
    transactions), and Symphony (execute-once-verify-many-times
    consensus protocol).

-   Ritual is proof-system agnostic and proof-system modular —
    developers choose ZKML, OPML, PPML, or TEE based on their security,
    cost, and performance requirements.

-   Infernet's 8,000+ node network is natively integrated into Ritual
    Chain, providing massive compute capacity for orchestrating AI
    workloads across any EVM-compatible blockchain.

-   Ritual serves as infrastructure for the broader web3 ecosystem —
    high-performance chains (MegaETH, Movement), RaaS platforms
    (Conduit, Caldera), and IP/model marketplaces (Story, Sentient) all
    benefit by building on or integrating with Ritual.
